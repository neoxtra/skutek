Reading GRETINA data into Python

To read Vireo Gretina files, SkuTek provides a python library called “skutils” which can parse the files and building events. This library can be downloaded from Vireo’s Web GUI. Navigate to the Programming Guide and click on the header link “Accessing Recorded Data”.

This library is intended to be installed on lab computers or servers used for offline processing. It can be installed a precompiled Python wheel file or used as Python source code.

The webpage section “Parsing Data Files” contains instructions for installation of the library and a link to additional usage documentation.

Once installed, the library can be imported in python as follows:

import skutils

Files can be opened using the GretinaLoader object. Events can be iterated through in a loop using the number next_event().

loader = skutils.GretinaLoader("path/to/your.geb")
metadata,event = loader.next_event()

Event data (‘event’ in the above example) is returned as a Pandas Dataframe. This is a tabular representation of the event data. The names of the columns are the channel index, and the rows are the samples for that channel.

Metadata (‘metadata’) in the above example is a python dictionary containing collection metadata such as the digitizer bitdepth, number of samples per trace, and pulse summary objects.

Refer to the full library documentation in the programming guide for more details.

File Format Specification

The Vireo can record data in the binary Gretina Event Builder (GEB) file format developed for the GRETA & GRETINA gamma ray detection arrays. This is a highly efficient format that saves data in a series of independently formatted packets.

These files are best decoded packet by packet in a while loop which reads packet headers, and then decodes the packet contents based on the information contained in the header. The rough approach is as follows. For each packet in file:

  1. Read header to determine packet type and size.

  2. Read the number of bytes specified by header into a buffer.

  3. Pass the buffer to a decoding function unique to the packet type.

  4. Decode packet contents.

All packets are prefixed with a standard Gretina Event Builder (GEB) header. GEB Headers are always 16 Bytes total and broken up as follows.

Size

Info

Packet Type

int32

Identifier for this type of packet. This should determine your decoding strategy.

Packet Length

int32

Size of this packet’s payload. (i.e. packet size NOT including this header).

Timestamp

int64

The timestamp for this packet.

In the form of a C struct, this is:

struct gretina_header {
   int32_t type; /// type of packet
   int32_t length; /// length of packet in bytes following this header
   int64_t timestamp; /// FPGA timestamp for this packet data
};

Packet length will always be a multiple of 4 bytes. 1-3 Bytes of null padding may be appended to the end of the packet to maintain this 4 byte alignment.

Endian (Byte Order) Packet

This is a header-only packet written at the beginning of Vireo data files. This will indicate the endianness (or order of bytes) for every 32bit (4byte) word. If you are decoding this file on a little-endian computer, then this packet can usually be safely ignored.

The entire packet following the header can be read into the following C structure.

struct Vireo_PulseSummaryPacket {
                           // member_size, total_size
   int32_t word1;        // 4, 4
   int16_t  pulse_height; // 2, 6
   int16_t  trig_height;  // 2, 8
   uint8_t  trig_count;   // 1, 9
   uint8_t  triggered;    // 1, 10
   uint16_t null_padding; // 2, 12
} __attribute__((__packed__, aligned(4) )); //Required to avoid compiler padding errors.
